# Second Brain Research Brief: LLM-Navigable Obsidian Vault

**Prepared by:** Pax  
**Date:** 2026-06-01  
**Audience:** Alyssa — planning session input  
**Decision:** How to structure, populate, and route the myPKA vault for daily LLM-driven content creation

---

## Executive Summary

Building for LLM navigation is a different design problem than building for human navigation — and most PKM advice targets humans. The structural choices that matter most for AI are: a machine-readable orientation layer (AGENTS.md / index files), strict separation of raw source material from synthesized wiki content, and consistent YAML frontmatter on every entity note. Alyssa's myPKA scaffold already implements several of these patterns correctly. The main gaps are: a master vault index (wiki-style index.md at the root level), explicit per-folder README/INDEX files LLMs can read as orientation, and a lint/maintenance cadence. The import side has a clear winner for each source type, with NotebookLM being the hardest and least reliable.

## Key Findings

### 1. What makes a vault LLM-navigable vs. human-navigable

**Confidence: High** (Obsidian Forum community thread, Karpathy LLM Wiki pattern, myPKA scaffold architecture, MindStudio documentation)

The core difference: humans navigate by visual scanning, muscle memory, and search. LLMs navigate by reading files in sequence. A vault designed for LLMs front-loads orientation — the model should know what the vault contains and where to look within the first 2-3 files it reads, without browsing.

Structural choices that favor LLM navigation:

- **A flat naming convention over deep nesting.** Folder depth beyond 3 levels creates navigation cost for LLMs. Concept folders should be flat with wikilinks doing the connecting work, not folder hierarchy.
- **YAML frontmatter on every entity note.** Fields like `type`, `status`, `tags`, `related`, and `created` are machine-readable in a way that prose bodies are not. LLMs can parse and filter on frontmatter without reading the full file. (This is what GL-002 in the myPKA scaffold enforces — already correct.)
- **An AGENTS.md or CLAUDE.md at the vault root.** This is the single most important orientation file. It tells the LLM: who the agents are, what the folder structure means, and what the rules are. myPKA already ships this. The discipline is keeping it current.
- **Per-folder INDEX.md files.** Every major folder should have a one-page README/INDEX that explains what lives there, what the naming convention is, and what a new LLM session needs to know before writing to that folder. These function as local AGENTS.md files for sub-areas.
- **A master vault index (wiki-style).** A single `index.md` or `HOME.md` at the root that catalogs all major areas, key files, and their one-line descriptions. The Karpathy LLM Wiki pattern treats this as non-negotiable — the model loads the index first, pulls specific pages as needed.

Alyssa's vault already has AGENTS.md and per-section INDEX.md files. The gap is a single master vault index that synthesizes across all sections in one place.

### 2. Multi-level map structures (Master Map → Folder Map → In-file TOC)

**Confidence: High** (LYT Kit / Nick Milo, Obsidian Forum, Dataview plugin documentation, Automatic Table of Contents plugin)

This three-level pattern is used by advanced vaults and is considered best practice in the LYT community. Here is how the levels function differently:

**Level 1 — Master Map (Home Note / Vault Index)**  
One file. Usually called `HOME.md`, `index.md`, or `000 - Home`. Lists every major section, MOC, and domain with a one-line description and a wikilink. This is orientation, not content. Nick Milo's LYT Kit ships a Home Note as the entry point for the entire vault. For LLM use, this file is also the "session start" file — the first thing any agent reads.

**Level 2 — Folder/Domain MOC**  
One per major topic area or folder. Lists every note in that area with a brief description. In LYT, MOCs are dynamic (human-maintained, updated when new notes appear). In Karpathy-style vaults, they are auto-generated by the LLM after each ingest. The Dataview plugin can auto-populate these if you want dynamic updates without LLM involvement. The distinction matters: human-curated MOCs reflect judgment; auto-generated MOCs reflect coverage. Both have value.

**Level 3 — In-file Table of Contents**  
For long notes (anything with 3+ H2 headers), a TOC at the top lets LLMs locate specific sections without reading the full file. The Automatic Table of Contents plugin handles this automatically. For LLM use, the more important practice is keeping H1/H2 headers semantic (descriptive, not clever), since LLMs skim headers before deciding whether to read the section.

**Verdict:** Alyssa's three-level thinking is correct and well-supported. The implementation note is that Level 1 is the highest-leverage investment and the most commonly skipped.

### 3. Import method comparison

**Confidence: High for Notion/local files; Medium for NotebookLM and web clipper; Low for "sending links directly to Claude" as a persistent import method**

#### Notion → Obsidian
Two official paths exist via the Obsidian Importer plugin:

| Method | Preserves Databases? | Requires API token? | Best for |
|---|---|---|---|
| File import (HTML export) | No | No | Simple pages, writing, notes |
| API import | Yes (as Bases) | Yes (ntn_ token, ~2 min setup) | Complex databases, relational content |

**Recommendation:** Use API import. The file import loses database structure, which is often where Notion's value lives. Notion API rate limits mean large workspaces take time — run it overnight. The official Obsidian Importer plugin is the most reliable path; third-party CLI tools (notion2obsidian on GitHub) are faster but require manual cleanup.

**Anti-pattern:** Exporting Notion as Markdown directly from Notion's export menu — Notion's own Markdown export omits important data. Use HTML export for file import, or use the API.

#### NotebookLM → Obsidian
NotebookLM has no native markdown/plain-text export as of June 2026. Its December 2025 export update only sends content to Google Docs or Sheets.

Workable paths:
- **Chrome extension (recommended):** "NotebookLM to LaTeX & MD" extension adds a download button that exports notes as Markdown. Drop directly into vault.
- **Copy-paste fallback:** NotebookLM's AI-generated reports use clean Markdown formatting. Copy-paste into Obsidian preserves headers and formatting. Not scalable but reliable for one-offs.

**Single-source flag:** The Chrome extension method is reported by XDA Developers as the most seamless path, but this is a single source. Treat as Medium confidence until Alyssa tests it.

**Anti-pattern:** Expecting NotebookLM to export structured knowledge in a repeatable way. It is a read-and-synthesize tool, not a knowledge base — the export problem is structural, not a bug they are fixing fast.

#### Obsidian Web Clipper (for web content)
The official Obsidian Web Clipper browser extension captures web pages directly into the vault with frontmatter, folder targeting, and template support. It is the cleanest path for building a permanent web-content library in Obsidian.

Strengths: Writes directly to vault with proper frontmatter, supports custom templates per content type, preserves metadata (author, date, URL, tags).  
Weakness: Optimized for human readability — keeps more page noise (nav, sidebars) than AI-focused clippers.

For LLM-optimized clipping, tools like Web2MD strip noise more aggressively and provide token counts for Claude/GPT context budget planning. The recommended stack: Obsidian Web Clipper for permanent vault additions; Web2MD (or pasting URL directly to Claude) for immediate analysis tasks.

#### Sending links directly to Claude
This is not a persistent import method — it is a session-scoped analysis method. Claude reads the URL, analyzes it, and the knowledge lives in that conversation only. Nothing persists to the vault unless you explicitly write an extraction from Claude's output.

**Verdict for Alyssa's use case:** Links sent directly to Claude should be followed by a structured extraction step — ask Claude to output a note in a defined frontmatter format, then save that note to the vault. This is a useful intake workflow, not a replacement for the web clipper.

#### Local file conversion
The most reliable import path. Local markdown, text, Word, or PDF files can be imported via the Obsidian Importer plugin or simply dropped into the vault folder. For non-markdown files, Pandoc handles conversion to clean Markdown at the command line. No API dependencies, no rate limits, no data loss risk.

### 4. Workstream / agent routing patterns

**Confidence: Medium** (Obsidian Forum thread, GitHub implementations: claude-obsidian, obsidian-second-brain, obsidian-wiki; Jin's Second Brain notes; MindStudio documentation)

The community has converged on three distinct agent roles for PKM systems, even when a single model plays all three:

**Role 1 — Ingestion Agent**  
Reads raw sources, extracts entities and concepts, creates or updates wiki pages, refreshes the index, logs the operation. The separation point is: this agent writes to the wiki layer, never to the source layer. Source files are immutable.

**Role 2 — Retrieval/Synthesis Agent**  
Answers queries by reading the wiki layer (not the raw sources). This is the "search my brain" role. Because it operates on pre-synthesized content rather than raw material, it produces denser, more useful answers.

**Role 3 — Maintenance Agent (Linter)**  
Scans the wiki for broken links, orphaned pages, duplicate concepts, stale claims, and missing cross-references. This role is most commonly skipped and most commonly the reason vaults degrade. Karpathy's pattern names this `/lint-wiki` and recommends running it on a schedule.

**How this maps to myPKA:** The scaffold already has these roles split across Penn (ingestion/capture), Pax (research/retrieval), and Silas (structural integrity). The linting role is currently session-close behavior for Larry. The main gap is making the lint operation explicit, scheduled, and documented as a workstream.

**Multi-agent orchestration patterns emerging in 2025:**  
- Scout + Analyst + Gatekeeper (ingestion crews): Scout finds sources, Analyst summarizes, Gatekeeper decides what gets filed.  
- Researcher + Writer + Editor (synthesis crews): for producing output documents from vault content.  
- These patterns are being implemented in CrewAI and LangGraph frameworks but are not yet standardized. Single-model multi-role (myPKA's approach) is more reliable at current tooling maturity.

### 5. Red flags and anti-patterns

**Confidence: High** (cross-verified across Obsidian Forum, Karpathy implementation guides, LYT Kit, multiple GitHub repositories)

**Anti-pattern 1 — Importing everything at once on day one.**  
The first few ingests need human supervision to calibrate how the LLM categorizes and cross-links content. Mass-importing 3 years of Notion data before validating the schema produces a large, poorly-structured vault that is harder to fix than starting small.

**Anti-pattern 2 — Skipping the maintenance loop.**  
Every LLM-navigable vault needs a lint/health-check operation. Without it, index drift (index.md listing pages that no longer exist, or missing new pages) accumulates until the LLM is navigating a stale map. This is the most common reason PKM systems fail after the initial setup enthusiasm fades.

**Anti-pattern 3 — Mixing raw sources and synthesized content.**  
Raw files (clipped articles, exported notes, voice memos) must live separately from synthesized wiki pages. If they are mixed, the LLM has no reliable signal about which content is curated vs. raw, and the quality of retrieval degrades. Keep a `Team Inbox/` (raw) and a `PKM/` (curated) layer — myPKA does this correctly.

**Anti-pattern 4 — Over-engineering the schema before validating the workflow.**  
Elaborate frontmatter schemas, complex Dataview queries, and multi-level tagging systems built before the basic ingest-retrieve-synthesize loop is working create overhead that kills adoption. Start with the minimum viable schema and add fields as actual retrieval gaps surface.

**Anti-pattern 5 — Clever note titles instead of semantic ones.**  
Note titles that are witty, abbreviated, or personal-shorthand-dependent cannot be navigated by an LLM that has not read every note. Semantic titles (descriptive, self-contained) allow the LLM to make routing decisions from the index without opening every file. This applies to headers within files too.

**Anti-pattern 6 — Using AI to generate notes without a review gate.**  
LLM-generated content that is filed directly into the knowledge base without human review introduces confident errors that are indistinguishable from verified knowledge. The Gatekeeper role in multi-agent patterns exists specifically to prevent this. In a solo vault, the human is the gatekeeper.

**Anti-pattern 7 — Treating NotebookLM as a knowledge base.**  
NotebookLM is a read-and-synthesize tool. Its value is in asking questions about uploaded sources, not in being a persistent knowledge store. Insights generated in NotebookLM need to be extracted and filed into Obsidian to have lasting value. Many users discover this 6 months in when they cannot find something they "know they put in NotebookLM."

## Recommendations

**Immediate (before planning session):**

1. **Add a master vault index.** Create `PKM/HOME.md` or a root-level `index.md` that lists every major PKM section, key MOCs, and what each folder contains in one sentence. This is the single highest-leverage structural addition for LLM navigation.

2. **Define the raw/curated separation explicitly.** Confirm that `Team Inbox/` is the only place raw unprocessed content lives, and that everything in `PKM/` is considered curated/verified. Document this rule in AGENTS.md.

3. **Start Notion import with the API method**, not file export. Accept that it will take time on a large workspace.

**For the planning session:**

4. **Decide the three-level MOC commitment level.** Level 1 (master index) is non-negotiable for LLM use. Level 2 (per-folder MOCs) — decide whether these will be human-curated or Dataview-auto-generated. Level 3 (in-file TOC) — use the Automatic Table of Contents plugin for any file over ~500 words.

5. **Build a lint workstream.** Define what "healthy vault" means (no broken wikilinks, index.md matches actual files, every entity note has required frontmatter fields). Schedule a monthly lint pass. Consider making this an explicit WS-NNN workstream in Team Knowledge.

6. **Set a Gatekeeper rule for LLM-generated content.** Any note produced by an LLM goes into `Team Inbox/` first, not directly into `PKM/`. Penn reviews and files. This prevents confident errors from polluting the knowledge base.
