1. What this repo is

KeepSidian is an Obsidian plugin, written in TypeScript, that brings your Google Keep notes into your vault and, in newer versions, pushes vault notes back up to Keep (two-way sync). The plugin itself does very little talking to Google directly. Almost all of the real work is done by a separate KeepSidian server (a Flask service based on Keep-It-Markdown). The plugin captures a Google login token, hands it to that server, and then asks the server for your notes. Your credentials are held on your machine, sent at sync time, and not stored on the server.

3. The core abstractions

KeepSidianPlugin (src/app/main.ts, 56 graph edges)

The center of the whole plugin. It extends Obsidian's Plugin class and, on load, builds the subscription service, registers the ribbon and commands, sets up the status bar and settings tabs, and starts auto sync if it is enabled. It also holds the live sync state: whether a sync is running, the progress modal, and the cached supporter status. Every user action (import, push, two-way) routes through a small guard here that checks credentials and prevents two syncs at once.

if (this.settings.autoSyncEnabled) {
  this.startAutoSync();
}
// ...
const intervalMs = this.settings.autoSyncIntervalHours * 60 * 60 * 1000;

The two-way gate (computeTwoWayGate in main.ts)

This is the client-side lock on uploads and two-way sync. It builds a list of reasons the action is blocked. You must have acknowledged the backup warning, enabled two-way in settings, and, when premium is required, have an active supporter subscription.

if (requirePremium && subscriptionActive === false) {
  reasons.push("To enable uploads & two-way sync, please consider becoming a KeepSidian supporter.");
}

The server calls (src/integrations/server/keepApi.ts)

Every real data operation is an HTTP call to the KeepSidian server, carrying your email and bearer token. There are two separate download endpoints: a standard one and a premium one. The premium call is the only place the AI features (suggested tags, suggested titles, note filtering) are requested, and they are sent to the server to run.

// standard:  GET  /keep/sync/v2
// premium:   POST /keep/sync/premium/v2   { feature_flags: { suggest_tags, suggest_title, ... } }
// push:      POST /keep/push
// register:  POST /register        (oauth token to keep_token)

SubscriptionService (src/services/subscription.ts)

A small class that asks the server /subscriber/info whether your email is an active subscriber, and caches the answer for 24 hours. This is what the two-way gate reads. It is a client-side check, so it decides what the plugin offers you, but it does not decide what the server will actually do.

4. How a typical sync works, start to finish

  1. Get a token once. You paste a Google oauth2_4... token. exchangeOauthToken posts it to the server's /register, which returns a long-lived keep_token. The plugin saves that token, redacting it in any logs.
  2. Open the Sync Center. The SyncProgressModal lets you review a plan (download, upload, or two-way) before anything changes. Credentials are checked first.
  3. Ask the server for notes. fetchNotes (or fetchNotesWithPremiumFeatures if you are a supporter) calls the server, which talks to Google Keep for you and returns notes as data.
  4. Validate and convert. The response is checked against a Zod schema, then each note runs through processAndSaveNote, which turns it into markdown and writes it into your vault at a safe, normalized path.
  5. Log and repeat. Each note is logged, progress is shown in the modal, and the loop pages through the rest. Two-way sync then compares and pushes changed vault notes back up through /keep/push.
The most useful thing this read confirmed: KeepSidian has two separate gates, not one. The supporter check inside the plugin (SubscriptionService plus computeTwoWayGate) only decides what the plugin offers you, which is why client-side things like the sync interval are freely adjustable. The AI features (auto-tagging, suggested titles) are a different story: they are only ever requested through the premium server endpoint and run on the server, so they cannot be unlocked from the plugin side. That matches exactly what you had noted: interval is client-side and free to change, auto-tagging is server-side and gated.
