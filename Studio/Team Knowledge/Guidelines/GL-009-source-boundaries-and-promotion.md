# GL-009 - Source Boundaries and Promotion

- **Status:** Active
- **Owner:** Larry + Silas
- **Applies to:** every agent, SOP, Workstream, Deliverables file, Second Brain edit, Hermes edit, and Team Knowledge edit.
- **Why this exists:** The team has been creating useful work that sometimes sits too close to Alyssa-original source. That breaks trust. Generated nuance can help the system, but it cannot become source by accident.

## The one-line rule

**Source is what Alyssa made. Everything else is a map, a lens, an output, or a task until promoted on purpose.**

## The four source classes

| Class | Definition | Examples | Can live where |
|---|---|---|---|
| `source` | Alyssa-original or externally canonical material. Her words, her products, her real business facts, real exports, real published assets. | Journal entries, voice notes, real emails, real carousels, BOH transcripts, offer pages, Notion product facts, screenshots/exports. | `Notebook/Journal/`, `PKM/Second Brain/examples/`, `PKM/Second Brain/programs/`, entity folders (`Notebook/Life/`), `Hermes/BUSINESS.md` snapshots when sourced. |
| `map` | A pointer layer that helps agents find source without digesting it. | INDEX files, COMPASS files, `Hermes/MAP.md`, routing tables, "open this when" notes. | Beside the source it routes, Team Knowledge, Hermes. |
| `derived` | Team analysis of source. Useful but not sacred. Must cite the source it came from and stay labeled as interpretation. | Analysis files, frameworks, templates, rubrics, forensics, synthesis notes. | `PKM/Second Brain/analysis/`, `frameworks/`, `templates/`, Hermes judgment/craft files, Team Knowledge when team-wide. |
| `output` | Work product from a session. It may be valuable, but it is not durable knowledge yet. | Reports, audits, plans, drafts, gate packages, handoffs, research briefs. | `Deliverables/` until promoted, dispatched, archived, or deleted. |
| `task` | A concrete open loop with owner, status, and next action. | Open task files. | `Studio/Team Knowledge/tasks/` — the ONE home per [[GL-013-the-house-rules]] Rule 3 (the Ready Queue was dissolved into it 2026-07-10 per Alyssa; see [[Archive/ready-queue-retired-2026-07-10]]). |

## Promotion rule

Before creating or promoting any file, run the source-boundary check:

1. Is this Alyssa-original source?
2. If not, is it a map that helps agents get to source faster?
3. If not, is it derived knowledge with clear provenance and a specific use?
4. If not, is it only an output that should stay in `Deliverables/`, become a task, or be deleted?

If the answer is unclear, do not add it to source areas.

## Mean Alyssa trust check

Before asking Alyssa, creating a new wiki note, editing Second Brain authority files, promoting an output, or claiming a source gap, ask:

- Would Alyssa wonder where this came from?
- Would she think this was fabricated?
- Would she say we should have read the full source first?
- Would she say this answer already exists in BOH, a freebie, past work, Google Drive, Notion, or old session logs?
- Would she be annoyed that this became a wiki note instead of a pointer, task, archive, or deletion?
- Did a later/better output make this one obsolete?
- Should external gold-standard proof be gathered before inventing a framework?

If any answer is yes, stop and fix the trust issue before proceeding.

## Versions Rule

When the remaining decision is taste, strategy, positioning, pricing nuance, personal truth, or another Alyssa-call, and the team cannot objectively prove one right answer, do not choose one version silently.

Present the two best versions:

- **Version A:** safer, closer to known source, lower risk.
- **Version B:** more nuanced, higher-upside, more original, or needing Alyssa's taste.

Alyssa chooses at this level. The team should not ask clerical questions that can be answered by reading source, searching past work, checking BOH/freebie/training material, searching old session logs, or running an existing SOP.

## Reconciliation rule

Every session-created artifact must leave the session with a fate:

- `promoted` - updated a source-adjacent map, derived file, SOP, Guideline, Workstream, Hermes rule, or task.
- `referenced` - linked from the correct map/index/task, without becoming source.
- `dispatched` - assigned to a specialist or Ready Queue item.
- `archived` - useful process artifact, not active knowledge.
- `deleted` - duplicate, false, superseded, or trust-breaking.
- `needs-Alyssa` - only for taste, strategy, pricing, personal truth, or source access.

No artifact may remain as "interesting."

Larry records this at close-session in the `## Daily knowledge metabolism` section from [[SOP-011-write-session-log]]. Weekly [[WS-006-weekly-review]] rolls those sections up into `## Grand metabolism`.

## Ponytail application

Apply [[GL-008-ponytail-lazy-senior-dev-philosophy]]:

1. Can we avoid creating a file?
2. Can we update an existing map?
3. Can we link to source instead of summarizing it?
4. Can this remain a Deliverables output until proven useful?
5. Only then create or promote the smallest file that preserves trust.
