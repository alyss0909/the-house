Repo understanding

adhd

A brainstorming method for AI. The name is just branding and has nothing to do with the medical condition. Source: uditakhourii/adhd.

1. What this repo is

This is a structured way to stop an AI from handing you the first obvious answer. The core belief, in the author's own words, is that the first three ideas a model gives you are the ideas any competent person would give in thirty seconds. Correct, but forgettable. The interesting answers live "in the awkward middle nobody walks into." So this tool makes the AI think many separate thoughts at once, each one from a deliberately strange point of view, then brings in a separate critical pass to score, sort, and develop the best of them. It ships in two forms that do the exact same thing: a skill you can drop into Claude with no install, and a small Node program with a command line tool for running it outside Claude or in bulk. The name "ADHD" is a metaphor for the wide, unfocused, many-directions-at-once first stage, followed by a focusing stage.

2. Architecture at a glance

The program is small and the pieces map cleanly to steps in the method.

The engine (engine.ts)

This is the conductor. It runs the whole two-phase process from start to finish and is the single most important file. The graph report calls its run() function the central hub, and that is correct.

The frames (frames.ts)

A list of fifteen "cognitive frames." Each one is a short instruction telling the AI to re-ask the problem from an unusual angle: as a hardware engineer, as a hostile competitor, as a curious ten-year-old, as a speedrunner looking for glitches, as someone with a zero-dollar budget and one hour.

The AI wrapper (llm.ts)

The thin layer that actually talks to the AI and cleans up its replies. It also holds the single most important design choice, explained below.

The renderer (render.ts) and the command line (cli.ts)

The renderer formats the final result for a terminal. The command line file is the front door you type into. The types file (types.ts) is just the shared vocabulary that keeps all the pieces agreeing on what an "idea" or a "score" is.

3. The core abstractions

run() in engine.ts, the conductor

Everything happens inside one function. It has three phases, and the comment at the top of the file states the rule that governs all of it: "Convergence happens after divergence, never during." Here is the first phase, where the AI fans out into many parallel thoughts at once:

// PHASE 1 — DIVERGE. Pure parallel fan-out. No branch sees another.
const branches = await Promise.all(
  frames.map((f) =>
    limit(async () => {
      const b = await divergeBranch(problem, context, f, ideasPerFrame, model);
      return b;
    }),
  ),
);

In plain terms: it takes the five chosen frames and asks the AI to brainstorm under each one at the same time, in separate isolated conversations. None of these conversations can see the others. That isolation is the whole trick, and it is enforced by the design rather than merely requested politely in the prompt.

The scoring rule, where the opinion lives

After all the ideas come back, a separate critical pass scores each one on three numbers from zero to ten: novelty (how far from the obvious), viability (could this actually work), and fit (does it answer the real question). Then it combines them into one number with fixed weights:

// Weight: novelty matters because the whole point is escaping the obvious,
// but viability is the gatekeeper — a brilliant unshippable idea is a trap.
const total = r.novelty * 0.35 + r.viability * 0.4 + r.fit * 0.25;

Viability carries the most weight. The author is saying that being clever is not enough, an idea has to be usable, otherwise it is a trap. And "trap" is a real thing in this system: if an idea looks attractive but has a hidden cost, the critic flags it with a one-line reason and it is pulled out of the running and listed separately as a warning.

The non-obvious pick, the signature move

Most tools would just hand you the highest-scoring idea. This one does something more interesting. After it has a shortlist of viable ideas, it deliberately surfaces the most novel one among them, even if it is not the top scorer, and marks it with a star:

// Non-obvious pick = highest novelty among the viable shortlist.
const nonObviousPick = [...shortlist].sort(
  (a, b) =>
    (b.score.novelty + b.score.viability * 0.5) -
    (a.score.novelty + a.score.viability * 0.5),
)[0];

This is the point of the whole tool made concrete. The safe answer is still in the list, but the design goes out of its way to also put a spotlight on the interesting-but-workable option you might otherwise skip past.

The frames, the source of the range

Each frame is a short paragraph that re-poses the question. For example the biology frame reads:

"Transplant a mechanism from biology — immune systems, neural
plasticity, cell signaling, evolution, gut flora — and force-fit it
onto this engineering problem."

Here is the full set of fifteen frames, straight from the code. This is the actual menu the tool draws from. Read it as a brainstorming deck you could point at any decision, not just a coding one.

  1. Hardware engineer. Think in latency, memory, and physical limits. Re-ask it as a hardware problem.
  2. Regulator or auditor. What here must be provable, traceable, or refusable?
  3. Ten-year-old. Someone who has never seen the thing before. Naive, unencumbered, ignores convention.
  4. Competitor trying to break it. Attack the obvious solution, find where it fails, then flip each attack into an idea.
  5. Biology. Steal a mechanism from nature (immune systems, evolution, cell signaling) and force-fit it.
  6. Logistics and supply chain. Borrow queues, batching, just-in-time, hub-and-spoke, last-mile.
  7. Game design. What are the loops, rewards, friction, save-states, shortcuts? Treat the person as a player.
  8. Markets. Buyers, sellers, auctions, futures, a clearing house. What does the trade look like?
  9. Inversion. Ask the opposite question. How would you guarantee the thing you do NOT want, then negate each answer back.
  10. Extreme: no money, one hour. The crudest version that still does the one load-bearing thing.
  11. Extreme: infinite budget, ten years. The maximalist version. What is only possible at that scale?
  12. Remove the load-bearing assumption. Name the thing everyone treats as fixed, imagine it gone, and see what opens up.
  13. Speedrunner. Glitches, skips, shortcuts. The abusive-but-legal path through the problem.
  14. Ant colony or swarm. No central planner. Many small agents, local rules. How does it solve itself?
  15. On-call at 3am. You get woken when this breaks. What design means you never get the call?

When the tool picks frames for a run, it always forces at least one "wild" frame into the mix so the brainstorm stays weird, and it biases toward practical engineering frames when the problem is code-shaped. The selection is shuffled every run, so asking the same question twice gives you different angles each time.

The one load-bearing choice, in llm.ts

This is the detail that makes the whole method real rather than just a nice prompt. Each of the parallel thoughts is a genuinely separate call to the AI, a fresh conversation with no memory of the others. The file says so directly:

// Each divergent branch is its own query() call so they run in true
// parallel — this is the "ADHD" fan-out. Branches don't see each other's
// output during divergence (mixing kills idea quality).

    // No tools — divergence is pure generation. Tools = convergence pressure.

Why this matters: if you instead asked a single AI to "list five different options," it would write option one, then read its own option one while writing option two, and everything would drift toward the first idea. That is called anchoring. By giving each frame its own sealed conversation, there is no first answer to anchor on, so the ideas come out genuinely different by construction rather than by hope.

4. How a typical flow works

Trace one real run of the command adhd "design a rate limiter that survives a leader election".

1
Problem comes in. The command line file (cli.ts) reads your question and options, then hands them to run() in the engine.
2
Frames are chosen. The engine asks frames.ts for five frames, weighted toward engineering angles because this is a code problem, always with at least one wild card.
3
Diverge in parallel. Five sealed AI conversations run at once, each brainstorming six ideas under its own frame, each explicitly told to generate and not to judge. The first three obvious ideas are banned.
4
Score and cluster. The critic wakes up. It scores every idea on novelty, viability, and fit, flags the traps, and groups the survivors into three to six clusters by their underlying angle so you can see the shape of the whole idea space.
5
Deepen the survivors. The top three non-trap ideas each get a fresh pass that sketches how they would actually work, names the biggest risk, names the first concrete step, and spins off three to five child ideas.
6
Render. render.ts prints it all in a fixed order: the wide set grouped by cluster, the shortlist with the starred non-obvious pick, the trap warnings, the deepened sketches, and finally one provocation, a single wildcard question to push into if nothing landed.
The whole run costs about ten AI calls and takes thirty to ninety seconds, roughly five to ten times a single answer. Both the skill and the library are honest about this and include a gate that tells the AI to skip the whole process for simple lookups or questions that have one canonical answer. It is meant for real decision points, not everyday questions.

5. Where to start if you want to extend it

  • src/engine.ts is the one file to understand first. The run() function is the entire method in one place, and the comments walk you through each phase.
  • src/frames.ts is where you would add or change the brainstorming angles. This is the easiest, highest-leverage place to customize the tool, and it needs no deep coding knowledge to read.
  • src/llm.ts holds the sealed-conversation design and the cleanup that rescues messy AI replies. Read it to see why the parallel isolation is real and not just promised.
  • src/types.ts is the shared vocabulary. Read it early because it names every concept (idea, score, trap, cluster, deepened idea) the other files pass around.
  • src/render.ts and src/cli.ts are the presentation and the front door. Change these if you want different output formatting or new command options.

What this means for Alyssa

Strip away the coding-shop packaging and this is a brainstorming engine with a genuinely good spine. It does three things worth stealing as a thinking process, not as software: it forces ideas from many deliberately different angles at once, it keeps the generating stage strictly separate from the judging stage, and it deliberately spotlights the interesting-but-workable idea instead of only the safe one.

That structure maps almost directly onto content ideation. "Diverge under many frames, then converge with a critic, then flag the non-obvious pick" is exactly how you would want to generate hooks, angles, or campaign concepts and avoid the flat, obvious first draft. The frame idea in particular is portable: re-asking "what should this carousel be about" as a ten-year-old, a hostile competitor, or a zero-budget hustler is a legitimate way to break out of the same five ideas every time.

The actual program does not fit a content business as-is. It is written for engineering problems (rate limiters, sharding, API design), it costs real money per run, and running the Node version means installing developer tools. You would not run this repo. But the method inside it is a strong candidate to rebuild in plain language as a brainstorming play for your own idea work, where the frames are content angles instead of engineering vantage points. The single most valuable idea to carry over is the hard wall between generating and judging: never let the critic into the room while ideas are still being born.
