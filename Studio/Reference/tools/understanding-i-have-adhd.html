Repo understanding

i-have-adhd

A skill that changes how an AI assistant writes back to you, so the answer is easy for a distractible brain to act on. Source: ayghri/i-have-adhd.

1. What this repo is

This is not an app and it is not a piece of software you run. It is a set of writing rules for an AI coding assistant, packaged so the assistant loads them automatically. When the rules are active, the assistant changes the shape of everything it says. It leads with the thing you should do next instead of a paragraph of background. It numbers steps. It gives you a real time estimate like "about 15 minutes" instead of "this will take some work." It never opens with "Great question" and never closes with "Hope that helps." The whole idea is that for a reader with ADHD, knowing the answer and actually doing the answer are two different things, and good formatting closes that gap. The real content of the whole project is one plain-English file of ten rules.

2. Architecture at a glance

There are only three moving parts, and they are simple.

The rulebook

One Markdown file, skills/i-have-adhd/SKILL.md, holds all ten rules plus the reasoning behind them and the cases where a rule should be broken. This is the entire product. Everything else in the repo just delivers this file to the assistant at the right moment.

The always-on switch

A tiny piece of code (hooks/always-on.js) and a settings file (hooks/hooks.json) let you turn the rules on permanently, so you do not have to type a command every session.

The quality test

A Python script (scripts/run_evals.py) and a folder of test cases (evals/) measure whether the rules actually make answers better, rather than just assuming they do.

The graph report for this repo confirms the shape. Its single most connected hub is "Install i-have-adhd" with 43 connections, which tells you most of the repo is about packaging and delivery. The actual substance lives in the two "Rules" clusters, and the evaluation harness sits off to one side as its own tight community. In plain terms: a small amount of real content, wrapped in a lot of careful delivery and testing.

3. The core abstractions

The ten rules (the heart of it)

The rules are written for an AI to obey, but they read like a plain style guide. A few, in the project's own words:

1. Lead with the next action
   Bad:  "Let's think about this. Your auth flow has a few moving pieces..."
   Good: "Run `npm install jsonwebtoken`, then edit `src/auth.ts:42`."

6. Give specific time estimates
   Bad:  "This will take some work."
   Good: "About 15 minutes if tests already cover this. An afternoon if not."

10. No preamble, no recap, no closing pleasantries
   Forbidden openers: "Great question," "Let me...", "Sure!"
   Forbidden closers: "Hope this helps," "Feel free to ask."

What makes this file more thoughtful than a normal "be concise" instruction is that it also says when to break the rules. If you ask it to "explain" or "walk me through," it drops the brevity and answers fully. If a destructive action is coming, it stops and confirms first. And it ends with a self-check: before sending, delete the first sentence if it just announces what you are about to do, and delete the last sentence if it asks "anything else?"

The always-on hook

This is the clever bit of packaging. Instead of asking you to invoke the skill every time, the project uses a "hook," which is a small script an assistant runs automatically at the start of a session. The whole script is about twenty lines. In plain English it says: if the user left a small marker file on their computer, read the rulebook and paste it into the session; otherwise do nothing and get out of the way.

const flagPath = path.join(claudeDir, '.i-have-adhd-always');

if (!fs.existsSync(flagPath)) process.exit(0);   // no marker, do nothing

const body = fs.readFileSync(skillPath, 'utf8')
  .replace(/^---\r?\n[\s\S]*?\r?\n---\r?\n/, ''); // strip the header

console.log('ADHD MODE ACTIVE (always-on)...\n\n' + body);

Two details worth noticing. First, the whole thing is wrapped so that if anything goes wrong it just exits quietly and never breaks your session. Second, turning the mode off is deliberately easy: you say "stop adhd mode" for one session, or delete that one marker file to turn it off for good. The design respects that a persistent behavior change needs an obvious off switch.

The evaluation harness

This is what separates the project from a casual blog post. The author does not just claim the style is better, they built a way to measure it. The Python script runs the same set of real questions two ways: once with a plain assistant ("baseline") and once with the rulebook applied ("candidate"). Each answer is then scored on five things, with correctness weighted highest:

WEIGHTS = {
    "correctness":   0.35,
    "autonomy":      0.25,
    "actionability": 0.20,
    "safety":        0.10,
    "concision":     0.10,
}

Concision is only worth ten percent. That is a deliberate statement: the point of the skill is not to be short, it is to be correct and act-on-able. Being brief is a minor bonus, and correctness matters more than three times as much. There is even a "release gate" that refuses to declare the styled version a winner unless it beats the plain version overall and does not lose ground on correctness or safety.

4. How a typical flow works

Here is the full path, start to finish, using the real files.

  1. You install the plugin. Packaging files (plugin.json, .claude-plugin/marketplace.json) tell the assistant this skill exists and what it does.
  2. You opt in to always-on by creating one empty marker file named .i-have-adhd-always in your assistant's config folder.
  3. You start a session. The settings file hooks/hooks.json tells the assistant to run always-on.js at session start.
  4. That script checks for your marker file. It finds it, reads the ten rules out of SKILL.md, and injects them into the session with a note that says the mode is active and how to turn it off.
  5. For the rest of the session, every answer the assistant gives is reshaped by those rules: action first, numbered steps, real time estimates, no fluffy opener or closer.
  6. Separately and offline, the author runs the evaluation script on a batch of test questions to confirm the styled answers actually score higher than plain ones before shipping a new version.
The elegant part: the assistant does not need to "understand" ADHD. It just receives a plain rulebook at the top of the conversation and follows it. The behavior change is entirely a matter of getting the right instructions in front of it at the right time.

5. Where to start if you want to extend it

  • skills/i-have-adhd/SKILL.md is where 95 percent of the value lives. If you wanted to adapt this to a different audience or a different voice, you would edit this one file and nothing else. It is plain English, not code.
  • hooks/always-on.js is the twenty-line delivery mechanism. Read it to understand how a skill turns itself on automatically without you typing anything.
  • hooks/hooks.json is the wiring that says "run that script at session start." Small, but it is what makes always-on possible.
  • scripts/run_evals.py plus the evals/ folder is the proof layer. If you wanted to test whether your edits to the rules made answers better or worse, this is the tool that measures it.
  • plugin.json and .claude-plugin/marketplace.json are the packaging labels. You only touch these if you are publishing or renaming the plugin.

What this means for Alyssa

This tool does one job: it forces an AI assistant to write in a way that is immediately act-on-able instead of chatty and padded. Lead with the thing to do, number the steps, cut the "Great question" and the "Hope this helps," give a real time estimate.

The interesting part for a content business is not the code, it is the rulebook itself. That single file is basically a distilled style guide for writing that respects a reader's attention. A lot of it lines up with rules you already enforce on your own surfaces: no fluffy openers, no empty closers, plain complete sentences, cut the hedging. If you ever wanted a house voice rule for "how the AI should hand me an answer so I can act on it fast," this file is a ready-made, well-argued template you could lift and rewrite in your own words.

It does not fit a content business as a piece of software to run. It fits as a piece of thinking to steal. The engineering around it (the hook, the eval harness) is built for a coding shop, and you would not use that machinery. But the ten rules, and especially the idea of scoring "did this actually help me act" above "was this short," are worth keeping.
